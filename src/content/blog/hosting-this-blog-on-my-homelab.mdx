This site is a set of static files in an object storage bucket on a machine in my apartment, served through a Cloudflare tunnel. Here's how it fits together and why I set it up this way.

## The pipeline

I push to `main`. A GitHub Actions runner on my own cluster picks up the job, builds the site with Astro, and syncs the output to a [Garage](https://garagehq.deuxfleurs.fr/) bucket. Garage serves that bucket as a website, and the tunnel points `mert574.dev` at it.

```bash
npm ci
npm run build            # astro build + pagefind index
aws s3 sync dist/ s3://mert574.dev/ \
  --endpoint-url http://192.168.178.109:3900 \
  --delete
```

The bucket is both the deploy target and the web server. Nothing runs per request, so there's less to keep patched and it comes back on its own after a reboot.

## Why the runner runs at home

Garage sits on my LAN, and the runner needs to reach it to push the build. I could expose the S3 endpoint through the tunnel, but Cloudflare's proxy caps request body size on the free plan, which breaks larger uploads. Running the job on a runner that's already on the LAN skips that problem, and I don't expose anything extra to do it.

## Why object storage

I didn't want a server to babysit. A static build in a bucket is cheap to serve and easy to reason about, and the backup is just the git repo plus whatever's in the bucket. Garage gives me S3-compatible storage and static website hosting in the same box, which is the whole job.

## What's still rough

The Kubernetes side of the cluster is already GitOps through Argo CD, but the NixOS hosts still need a manual apply when I change their config. Closing that gap, so host changes roll out on their own, is the next thing I want to sort out.
